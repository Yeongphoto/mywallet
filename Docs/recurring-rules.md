# 정기기록 규칙

## 핵심 원칙

- 정기기록 ON/OFF는 거래 자체를 복잡하게 변환하는 기능이 아니다.
- 정기기록 ON이면 다음 달부터 자동 기록 규칙이 정확히 작동하면 된다.
- 정기기록 OFF이면 다음 달부터 자동 기록이 중단되면 된다.
- 아래 정기기록 뷰는 등록된 반복 규칙을 확인하고 관리하는 화면이다.

## 거래 등록 시 정기기록 ON

- 현재 등록하는 거래는 해당 월의 실제 거래로 저장한다.
- 동시에 반복 규칙을 생성한다.
- 반복 규칙의 시작 월은 현재 거래 월의 다음 달이다.
- 다음 달부터 같은 일자, 같은 구분, 같은 카테고리, 같은 내용, 같은 금액으로 자동 기록된다.

## 거래 수정 시 정기기록 ON

- 현재 거래의 날짜, 내용, 금액, 카테고리 수정은 그대로 저장한다.
- 정기기록 ON을 켜면 현재 수정값을 기준으로 반복 규칙만 켠다.
- 현재 거래를 자동 생성 거래처럼 바꾸지 않는다.
- 현재 거래 ID를 임의로 `rec_...` 형태로 바꾸지 않는다.
- 반복 규칙의 시작 월은 현재 거래 월의 다음 달이다.

## 정기기록 OFF

- 현재 거래는 그대로 남긴다.
- 연결된 반복 규칙만 중단 처리한다.
- 중단 기준은 해당 거래 월이다.
- 다음 달부터 자동 기록이 생성되지 않아야 한다.

## 정기기록 뷰

- 정기기록 뷰는 반복 규칙을 보여주는 관리 화면이다.
- 진행 중인 규칙은 정상 상태로 표시한다.
- 끊긴 규칙은 비활성 상태처럼 회색 처리한다.
- 삭제를 누르면 정기기록 뷰 목록에서 제거한다.
- 기존 거래 내역은 규칙 삭제만으로 삭제하지 않는다.

## 작업 원칙

- 사용자가 말한 범위 밖으로 정책을 바꾸지 않는다.
- 문제 원인을 추측해서 기능 방향을 바꾸지 않는다.
- 질문받은 지점에 대해 코드와 현재 동작을 확인한 뒤 답한다.
- 수정은 사용자가 요청한 문제 범위 안에서만 한다.
- 정책 변경, 데이터 기준 변경, D1/로컬 우선순위 변경이 필요해 보이면 먼저 사용자 승인을 요청한다.
- 특히 D1 저장 정책은 임의로 바꾸지 않는다.
- 한국어가 포함된 파일은 UTF-8 인코딩을 유지한다.
- 터미널 출력에서 글자가 깨져 보이더라도 실제 파일 인코딩을 확인하기 전에는 텍스트를 재작성하지 않는다.
- 기존 한국어 UI 문구를 수정할 때는 인코딩 손상 없이 필요한 문구만 정확히 바꾼다.

## UI 정리 원칙

- 레이아웃 정렬, 버튼 크기, 디자인, 화면 밸런스는 개별 요소만 보지 않고 전체 화면 기준으로 맞춘다.
- 버튼은 같은 영역 안에서 높이, 폭, 여백, 정렬 기준을 통일한다.
- 카드, 하단바, 모달, 설정 영역은 PC와 모바일에서 각각 실제 화면 비율을 기준으로 균형을 맞춘다.
- 불필요한 설명 문구나 장식보다 조작 위치, 가독성, 간격, 터치 영역을 우선한다.

## 저장 로직 원칙

- 저장 로직은 최우선 안정 영역이다.
- 저장 로직 수정은 사용성 문제를 만들면 안 된다.
- D1을 기준 저장소로 바라보는 정책을 임의로 바꾸지 않는다.
- 로컬 저장은 보조 안전장치로만 다룬다.
- 로컬 데이터를 근거로 D1 우선 정책을 뒤집지 않는다.
- 오래된 데이터, 누락된 payload, 일부 필드만 포함된 payload가 D1의 정상 데이터를 지우는 일이 없어야 한다.
- 거래, 자산, 계획, 카테고리, 정기기록, 삭제된 정기거래 ID, 설정값은 저장/복원 과정에서 함께 보존되어야 한다.
- 저장 로직을 고쳐야 할 경우 먼저 원인, 영향 범위, 변경 방향을 보고하고 승인을 받은 뒤 수정한다.
- 저장 관련 변경 후에는 최소한 빌드 확인과 D1 저장 payload 범위 확인을 한다.
