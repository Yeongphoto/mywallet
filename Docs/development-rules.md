# 개발 규칙

## 인코딩 / 개행 안전 규칙

- 일본어를 포함한 기존 파일은 읽기 전에 encoding 후보, BOM 유무, 개행 코드를 확인한다.
- 문자 깨짐이 의심스러운 파일은 확신이 들 때까지 저장하지 않는다.
- 기존 파일은 원래의 encoding, BOM, 개행을 유지한다.
- 새 파일은 리포지토리 규약에 따라 UTF-8 계열로 통일한다.
- 쓰기는 encoding을 명시할 수 있는 방법만 사용한다.
- 저장 후에는 재독해 일본어의 대표 행을 검증한다.
- 한글/일본어/비ASCII 데이터를 API로 보낼 때 PowerShell 객체/파이프라인 경유 저장을 금지한다.
- 비ASCII payload는 UTF-8 파일 기반 Node.js 스크립트 또는 브라우저 fetch처럼 UTF-8 보존이 확인된 경로만 사용한다.
- DB 전체 저장 API를 임시 검증 목적으로 호출하지 않는다. 필요한 테이블/레코드만 저장하는 전용 API를 사용한다.
